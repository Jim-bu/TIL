## 🔵 Tech → Product (기술에서 서비스로)

**firecrawl(크롤링 인프라) → AnySearch(실시간 구조화 검색)**

에이전트가 고도화될수록 학습된 과거 지식이 아니라 **실시간 웹 데이터**를 필요로 한다. `firecrawl`은 이 수요를 정확히 겨냥한 고성능 분산 스크래핑 인프라로, SPA를 렌더링해 대기했다가 필요한 리소스만 추출하고 HTML DOM을 LLM이 먹기 좋은 클린 Markdown으로 정제한다.

이 기술 계층 위에 상용 서비스를 얹으면 바로 **AnySearch**가 된다 — "에이전트가 신뢰하는 실시간 구조화 검색". 즉 *크롤링 엔진(firecrawl류) → 프록시 로테이션으로 IP 차단 회피 → 수집 HTML을 토큰 효율 위해 Markdown 1차 정제 → JSON Schema로 구조화 → 저지연 캐싱*이라는 파이프라인을 SaaS API로 포장한 것. **오픈소스 인프라가 어떻게 상용 API 제품으로 전환되는가**를 보여주는 전형적 사례다.

## 🟢 Product → Tech (서비스에서 기술로)

**Typeahead 2.0(프라이빗 자동완성) → meetily의 On-device 아키텍처**

**Typeahead 2.0**는 "클라우드로 안 보내는 프라이빗 AI 자동완성"을 내세운다. 여기서 역으로 추적하면: 클라우드 전송이 없다는 건 **온디바이스 소형 모델(SLM) 추론**을 쓴다는 뜻이고, 모든 앱에서 동작한다는 건 **OS Accessibility API 후킹 + 키보드 이벤트 스트림**을 가로챈다는 뜻이다. 관건은 3B~8B급 모델을 **양자화(INT4/FP16)** 하고 **CoreML / llama.cpp**로 메모리 누수 없이 서빙하는 것.

이 "프라이버시-우선 로컬 추론" 아키텍처의 실제 구현체가 GitHub의 **meetily**다 — 클라우드 없는 로컬 회의 비서. 소스를 까보면 **Tauri(Rust + Next.js)** 위에서 **Whisper.cpp/Parakeet(ONNX)** 로 로컬 STT를 돌리고 **Ollama**로 로컬 요약을 한다. Typeahead가 텍스트에서 하는 일을, meetily는 오디오에서 하는 셈 — **같은 온디바이스 추론 패턴의 두 변주**로 묶어서 공부하면 된다.

## 💡 Deep Dive — 오늘의 인사이트

- **🧠 주요 개념** — *에이전트를 위한 인프라(Picks & Shovels).* 오늘 뜬 것들은 화려한 최종 앱이 아니라, 에이전트가 굴러가게 해주는 **데이터 공급(firecrawl)** 과 **로컬 추론(meetily·Typeahead)** 계층이다. "골드러시엔 곡괭이를 팔아라" — 에이전트 시대의 곡괭이가 지금 상위권을 채우고 있다.
- **📦 주목 프로덕트** — *AnySearch.* "사람이 쓰는 검색"이 아니라 **"에이전트가 쓰는 검색"** 으로 포지셔닝한 게 핵심. 같은 크롤링 기술이라도 소비자를 사람→에이전트로 바꾸자 완전히 새 시장이 열렸다.
- **🎯 핵심 기획력** — *제약을 세일즈 포인트로 전환.* Typeahead 2.0은 "온디바이스라 느리다"는 약점을, **"클라우드로 안 보낸다(프라이버시)"** 는 강점으로 재프레이밍했다. 기술적 한계를 사용자 가치로 번역하는 기획이 승부처.

**한 줄 종합:** 오늘의 승부는 *더 똑똑한 AI*가 아니라, **"누가 쓰는가(사람 vs 에이전트)"와 "어디서 도는가(클라우드 vs 온디바이스)"를 다시 정의한 포지셔닝**에서 갈렸다.
