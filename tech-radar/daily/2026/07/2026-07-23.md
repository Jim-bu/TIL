## 🔵 Tech → Product (기술에서 서비스로)

**Confidential Containers / microVM(격리 인프라) → box("에이전트용 풀 VM")**

CNCF에서 **Confidential Containers**가 인큐베이팅으로 승격했다 — 실행 중(in-use) 데이터까지 하드웨어로 격리하는 기밀 컴퓨팅. 여기에 **microVM(Firecracker)** 계열의 빠른 부팅·스냅샷을 더하면 "가볍지만 강하게 격리된 실행 환경"이 된다.

이 격리 primitive를 에이전트용으로 포장하면 **box**("에이전트를 위한 간단한 컴퓨터 = 풀 VM")가 된다. 에이전트가 마음껏 코드를 실행하되 **폭발 반경을 VM 경계로 가두는** 제품. 즉 *microVM/기밀 컨테이너(격리 기술) → per-agent VM(제품)*. box 같은 제품의 하부를 이해하려면 Firecracker·gVisor·Confidential Containers의 격리 모델을 보면 된다. (07-08 CubeSandbox의 계보.)

## 🟢 Product → Tech (서비스에서 기술로)

**Kastra("Claude/Cursor/Codex 런타임 인가") → 정책 엔진 + 에이전트 신원**

**Kastra**는 "코딩 에이전트를 위한 런타임 authorization"이다. 역으로 뜯으면: ① 에이전트에 **신원(identity)** 부여, ② 에이전트의 **각 툴콜/액션을 실시간 정책 평가**(누가·무엇을·어느 리소스에), ③ 승인/거절 + **감사로그**. 이건 K8s에서 워크로드가 클러스터에 들어가기 전 **Kyverno/OPA가 정책으로 막는** 패턴을 *에이전트 액션* 에 그대로 적용한 것.

그래서 오늘 CNCF의 *"정책 엔진을 프로덕션인 척 속였다(Kyverno)"* 글이 Kastra의 하부 그림과 정확히 겹친다. 또 07-13 `destructive_command_guard`(위험 명령 차단)·07-15 Agentcard(지출 승인 웹훅)와 같은 계열 — **자율 행동에 정책 브레이크**. Kastra를 이해하려면 정책-as-code(OPA rego/Kyverno)와 워크로드 신원(OIDC) 구현을 보면 된다.

## 🟣 Tech → Business (기술에서 비즈니스로)

**"엔터프라이즈 에이전트, 거버넌스 인프라가 못 따라간다" → 통제 평면(authz·격리·신원)을 쥐는 쪽이 예산을 먹는다**

The New Stack은 *Claw 스타일 자율 에이전트가 엔터프라이즈로 오는데 **거버넌스 인프라가 아직 못 따라간다*** 고 짚는다. 주목할 건 파트너 명단 — **Automation Anywhere + Cisco(네트워크)·Nvidia(격리/컴퓨트)·Okta(신원)·OpenAI(모델)**. 즉 "에이전트 거버넌스 스택"이 *신원(Okta) + 격리(Nvidia/기밀컴퓨팅) + 네트워크(Cisco) + 정책* 으로 짜이는 중이다.

결론: 엔터프라이즈가 자율 에이전트를 원할수록 **병목이자 지출처는 통제 평면**(런타임 authz·격리·신원·감사)이다. 모델·에이전트 자체가 아니라 이를 *안전하게 가두고 감사*하는 계층이 매출과 파트너십의 중심이 된다(오늘 PH의 Kastra·box가 딱 그 계층의 스타트업 버전). 또 TNS *"Crafting: 에이전트가 코드를 짜면 배포는 우리가"* 처럼 가치가 **코드 생성 → 배포·거버넌스**로 이동. 스타트업 시사점: **통제 평면(authz/격리/배포)** 에 서면 인수·파트너십·예산의 대상이 된다.

## 🔬 Enterprise Deep Dive — 에이전트 런타임 인가 + 격리 실행 (Kastra · Kyverno/OPA · microVM)

> 오늘의 1 Pick. "에이전트에게 권한을 주되, 액션마다 심사하고 폭발 반경을 가둔다."

### 왜 쓰나 (도입 근거)
- **해결 문제**: 에이전트에 정적 API 키/올오어낫싱 권한을 주면, 오작동·프롬프트 인젝션 하나가 **프로덕션 파괴/데이터 유출**로 직결. 최소권한 + **액션별 인가** + **격리**가 없으면 프로덕션 승인이 안 난다(TNS: 거버넌스가 병목).
- **대안 대비**: *정적 권한*(간단하지만 과잉권한) ↔ *런타임 authz*(각 툴콜을 정책 평가) + *격리 실행*(microVM/기밀 컨테이너로 blast radius 제한).
- **도입 효과**: 최소권한 강제, 전 액션 감사추적(규제 대응), 사고 시 폭발 반경 제한.
- **비용/리스크**: 정책 과도 → 에이전트 무력화(균형 필요), 신원 위조, 감사로그 폭증.

### 어떻게 연동하나 (기술 통합)
- **스택 위치**: `에이전트 툴콜` → `authz 미들웨어(Kastra류)` → *정책 통과 시* `격리 샌드박스(microVM/gVisor/Confidential Containers)` 에서 실행 → `감사로그`.
- **최소 통합(개념 예시)**:
```rego
# OPA/Kyverno류 정책: "prod DB엔 읽기만, 결제는 한도 이하만"
package agent.authz
default allow = false
allow {
  input.action == "db.query"; input.mode == "read"; input.env == "prod"
}
allow {
  input.action == "payment.charge"; input.amount <= data.limits[input.agent]
}
```
```bash
# 승인된 액션만 격리 VM에서 실행 (개념)
firecracker --config vm.json   # per-agent microVM, 스냅샷 복원으로 빠른 기동
```
- **운영 고려**: 에이전트 **신원**(워크로드 OIDC), 정책은 코드로 버전관리·테스트, 격리는 워크로드 민감도별(microVM ↔ 기밀 컨테이너), 감사로그는 07-16 OTel로 수집.
- **함정(gotchas)**: ① 정책 공백=암묵 허용 위험(default deny), ② 프롬프트 인젝션으로 권한 오용, ③ microVM cold start, ④ 사이드채널(→기밀 컴퓨팅으로 완화).

### 도입 판단 체크리스트
- [ ] 에이전트가 **쓰기/실행/결제** 권한을 갖는가?
- [ ] 현재 권한이 **올오어낫싱/정적 키**인가?
- [ ] **멀티테넌시**(여러 팀/고객) 격리가 필요한가?
- [ ] 규제·감사로 **전 액션 추적**이 필요한가?

→ 2개 이상 Yes면 **default-deny 정책 + 액션별 authz + 격리 실행**을 도입. 첫 수는 "정적 키 제거 + 액션별 최소권한 정책".

## 💡 Deep Dive — 오늘의 인사이트

- **🧠 주요 개념** — *Runtime Authorization & Isolation.* 에이전트 보안이 "무엇을 아는가"가 아니라 **"각 액션을 어떻게 인가·격리하는가"**로. Kastra(authz)·box(VM)·Confidential Containers(기밀)·Kyverno(정책)가 같은 층을 채움.
- **📦 주목 프로덕트** — *Kastra.* "코딩 에이전트용 런타임 인가"라는 포지션 자체가, 에이전트가 이미 **위험한 권한을 쥐고 있다**는 시장 인식을 방증.
- **🎯 핵심 기획력** — *통제 평면을 판다.* box·Kastra는 화려한 에이전트가 아니라 **에이전트를 가두는 인프라**를 팔아, "에이전트를 프로덕션에 못 올리는" 진짜 통증을 정조준.
- **💼 Tech → Business** — 거버넌스 스택이 **신원(Okta)+격리(Nvidia)+네트워크(Cisco)+정책** 으로 형성. 해자·예산이 모델이 아니라 **통제 평면**으로 이동, 인수·파트너십도 그쪽에 몰림.
- **한 줄 종합:** 어제 "조직 운영·거버넌스"에서 한 겹 더 내려가, 오늘은 **"에이전트를 액션 단위로 인가하고 VM으로 가두는" 보안 배관** — 자율성이 커질수록 브레이크·격리가 곧 제품이자 해자가 된다.
