## 🔵 Tech → Product (기술에서 서비스로)

**kvcache-ai/ktransformers(추론 최적화 프레임워크) → BaseRT(고속 로컬 런타임 제품)**

`ktransformers`는 이기종 환경에서 LLM 추론을 빠르게 하는 최적화 기법의 집합이다 — **KV cache 관리, GPU/CPU 오프로딩, 양자화·커널 튜닝**. 즉 "같은 모델을 어떻게 더 싸고 빠르게 돌리나"의 연구·구현체.

이 최적화를 **드롭인 런타임 제품**으로 포장하면 **BaseRT**("llama.cpp보다 6.4x, MLX보다 3.9x 빠름")가 된다. Apple Silicon에 특화해 커널·메모리 레이아웃·양자화를 갈아넣은 것. ktransformers류의 최적화 코드를 읽으면, BaseRT가 "무엇을 갈아서 그 속도를 냈는지"(배칭·KV cache·양자화 경로)가 그려진다. **최적화 연구 → 상용 런타임**의 전형.

## 🟢 Product → Tech (서비스에서 기술로)

**Rewisp("한 번 보면 영원히 물어본다") → 온디바이스 앰비언트 메모리 + 로컬 인덱싱**

**Rewisp**는 "본 것을 나중에 언제든 물어본다"는 Mac용 앰비언트 메모리다. 역으로 뜯으면: ① 화면을 **상시 캡처** → ② OCR/비전으로 텍스트·요소 추출 → ③ **임베딩 + 로컬 벡터 저장** → ④ 질의 시 의미 기반 회상. 관건은 프라이버시(전송 없이 **로컬에서** 캡처·인덱싱)와 비용(상시 캡처의 저장·전력).

이 "로컬-퍼스트 인덱싱" 패턴은 오늘 GitHub의 **code-review-graph**(로컬 코드 그래프)·**wigolo**(로컬 웹 리서치)와 정확히 같은 계열이다 — 데이터를 클라우드로 보내지 않고 **로컬에 지속 인덱스를 쌓고 MCP로 에이전트에 노출**. Rewisp의 회상 계층을 이해하려면 이 로컬 벡터·그래프 인덱싱 구현을 보면 된다. (07-13 "메모리" 테마의 온디바이스 버전.)

## 🟣 Tech → Business (기술에서 비즈니스로)

**"AI가 플랫폼 엔지니어링에 흡수된다" → 지출·해자가 내부 플랫폼팀으로 이동**

The New Stack의 세 기사(*AI가 플랫폼 엔지니어링과 병합 / 에이전트 속도로 환경 공급 / AI 클라우드 6개 카테고리*)가 한 방향을 가리킨다: 기업은 더 이상 **관리형 API만 부르지 않는다.** Blackwell/GB200發 GPU 옵션과 전문 프로바이더가 쏟아지면서, **자체·하이브리드 추론**(vLLM on K8s + HAMi GPU 공유)이 비용·데이터주권·지연 측면에서 현실적 선택지가 됐다.

결과적으로 AI 지출과 통제권이 **플랫폼 엔지니어링 팀**으로 이동한다 — 구매 주체가 "앱 개발자"에서 "내부 플랫폼팀"으로 바뀌고, 해자는 *모델*이 아니라 **모델·에이전트를 '에이전트 속도'로 서빙하는 내부 플랫폼**이 된다. 벤더들이 앞다퉈 "에이전트 플랫폼"(Google/Microsoft)과 "데이터 컨텍스트"를 내세우는 이유다. 스타트업 시사점: **플랫폼팀을 고객으로, 자체호스팅 스택에 sticky하게 박히는** 제품이 예산을 확보한다.

## 🔬 Enterprise Deep Dive — Kubernetes 위 자체 호스팅 LLM 추론 (vLLM + GPU 공유)

> 오늘의 1 Pick. CNCF(vLLM·HAMi) + TNS(플랫폼 엔지니어링) + GitHub(ktransformers) + PH(BaseRT)가 모두 가리키는 지점.

### 왜 쓰나 (도입 근거)
- **해결 문제**: 관리형 API는 편하지만 ① 토큰 비용이 규모에 비례해 폭증, ② 데이터가 외부로 나감(규제/기밀), ③ 모델·버전·지연을 통제 못 함.
- **대안 대비**: *완전 관리형 API*(간편·빠른 시작, 대신 비용·락인·데이터유출) ↔ *자체 호스팅*(운영부담↑, 대신 단가↓·데이터 사내·커스터마이즈). **vLLM**은 PagedAttention·연속 배칭으로 **높은 처리량/낮은 단가**를 내는 서빙 엔진.
- **도입 효과**: 대량·반복 워크로드의 **추론 단가 급감**, 데이터 사내 유지(규제 충족), 파인튜닝·양자화로 자사 특화.
- **비용/리스크**: GPU 확보·유휴 비용, 운영 복잡도, cold start, 멀티테넌시 격리.

### 어떻게 연동하나 (기술 통합)
- **스택 위치**: `클라이언트` → `vLLM 서버(OpenAI 호환 API)` → `K8s Deployment(GPU 노드)` → `HAMi(GPU 분할공유)` + `HPA/KEDA(오토스케일)` + `OTel(관측)`.
- **최소 통합(개념 예시)**:
```bash
# vLLM: OpenAI 호환 서버 한 줄로 기동
vllm serve meta-llama/Llama-3.1-8B-Instruct --max-model-len 8192
# 앱은 base_url만 바꿔 그대로 사용 (마이그레이션 최소)
#   OpenAI(base_url="http://vllm-svc:8000/v1")
```
```yaml
# K8s: GPU 요청 + HAMi로 한 장을 여러 파드가 분할 사용
resources:
  limits:
    nvidia.com/gpu: 1          # HAMi 설정 시 분수 GPU/메모리 지정 가능
```
- **운영 고려**: KV cache 메모리로 동시성 결정 → `--max-model-len`·배치 튜닝. 유휴 GPU 비용 → KEDA로 0-스케일. 관측은 앞서 07-16의 OTel(`gen_ai.*`)로 토큰·지연 추적.
- **함정(gotchas)**: ① GPU 메모리 초과 시 OOM/스로틀, ② 모델 로딩 cold start(수십 초) → 워밍/상시 파드, ③ 여러 팀 공유 시 격리·쿼터(HAMi 없인 GPU 독점).

### 도입 판단 체크리스트
- [ ] 월 관리형 API 비용이 **자체 GPU 운영비를 넘거나 근접**하는가?
- [ ] 프롬프트/데이터에 **규제·기밀**이 있어 외부 전송이 곤란한가?
- [ ] 안정적 **지연/가용성**을 자체 통제해야 하는가?
- [ ] **GPU·K8s 운영 역량**(또는 매니지드 K8s+GPU)이 있는가?

→ 앞 3개 중 2개 이상 Yes면 자체 호스팅 검토, 마지막 항목이 No면 우선 매니지드 GPU/K8s부터.

- **🧠 주요 개념** — *Inference Ops(추론 운영).* 경쟁축이 "어떤 모델이냐"에서 **"그 모델을 얼마나 싸게·빠르게·사내에서 돌리느냐"**(vLLM·BaseRT·ktransformers·GPU 공유)로 이동.
- **📦 주목 프로덕트** — *BaseRT.* "6.4x 빠르다"는 벤치마크 한 줄로 가치를 못박음. 추상적 최적화를 **측정 가능한 속도**로 번역해 개발자 지갑을 여는 전형.
- **🎯 핵심 기획력** — *로컬-퍼스트를 프라이버시 자산으로.* Rewisp·wigolo·code-review-graph 모두 "클라우드로 안 보낸다"를 전면에. 제약(로컬)이 곧 신뢰 세일즈 포인트.
- **💼 Tech → Business** — AI가 **플랫폼 엔지니어링에 흡수**되며, 지출·해자가 *모델*에서 *자체호스팅 서빙 플랫폼(플랫폼팀 소유)* 으로 이동. 구매 주체가 바뀐다.
- **한 줄 종합:** 지난주가 "에이전트를 어떻게 운영·소유하나(거버넌스·M&A)"였다면, 이번 주 시작은 **"모델을 어떻게 사내에서 싸고 빠르게 돌리나(자체 추론·로컬-퍼스트)"** — AI가 애플리케이션 레이어에서 **인프라·플랫폼 레이어로 내려앉는** 중.
