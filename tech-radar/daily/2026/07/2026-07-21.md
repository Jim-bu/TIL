## 🔵 Tech → Product (기술에서 서비스로)

**topoteretes/cognee(self-hosted 메모리 그래프) → Skippr AI(제품 속 AI 직원)**

`cognee`는 데이터를 인제스트해 **엔티티·관계를 지식 그래프 + 벡터**로 쌓고, 에이전트가 이를 회상하게 해주는 메모리 플랫폼이다. 핵심은 단순 문서검색(RAG)을 넘어 "무엇이 무엇과 어떻게 연결되는지"를 유지하는 것.

이 메모리 계층 위에 상용 서비스를 얹으면 **Skippr AI**("제품 안에 상주하며 모든 유저를 응대하는 AI 직원")가 된다. 인앱 AI 직원이 정확히 답하려면 *제품 문서 + 각 유저의 상태·이력*을 지속 기억해야 하는데, 그게 바로 cognee류 메모리 그래프다. 파이프라인: *제품·유저 데이터 인제스트 → 지식그래프+벡터 → 에이전트가 그라운딩해 응대*. **메모리 인프라 → 인앱 에이전트 제품**의 전형.

## 🟢 Product → Tech (서비스에서 기술로)

**Nautis("파운더용 AI-native OS") → 통합 허브 + 오케스트레이션 + 컨텍스트 그래프**

**Nautis**는 "파운더를 위한 AI 네이티브 OS"를 표방한다. 역으로 뜯으면 세 층: ① 이메일·문서·CRM 등 **도구 통합 허브**, ② 작업을 적절한 모델/에이전트로 보내는 **오케스트레이션/라우팅**, ③ 도구 경계를 넘는 **공유 컨텍스트 그래프**. 진짜 난도는 ③ — Microsoft가 말한 "데이터 컨텍스트" 문제다.

그 하부 구현이 오늘 GitHub에 다 있다: 컨텍스트는 **cognee/code-review-graph**(지식·코드 그래프), 라우팅은 **OmniRoute**(268+ 프로바이더). 즉 "AI OS"라는 제품은 *통합 커넥터 + 라우팅 게이트웨이 + 지속 컨텍스트 그래프*의 조립이며, 이 오픈소스들을 보면 Nautis가 어떤 부품으로 서 있는지 그려진다.

## 🟣 Tech → Business (기술에서 비즈니스로)

**"에이전트가 엔터프라이즈 OS로 변모" → 해자는 모델이 아니라 '조직 컨텍스트'와 통합 깊이**

The New Stack의 세 기사(*에이전트=엔터프라이즈 OS / IaC→플랫폼→에이전트로 운영 스택 진화 / Microsoft: 데이터 컨텍스트가 관건*)가 한 결론을 가리킨다: 엔터프라이즈 가치가 **모델**에서 **조직의 지속 컨텍스트(메모리·지식그래프)와 통합 깊이**로 이동한다. 에이전트가 조직의 운영 계층이 되려면 "우리 회사가 어떻게 돌아가는지"를 기억해야 하고, 그 컨텍스트를 소유·통제하는 쪽이 플랫폼을 쥔다(Microsoft의 data fabric 베팅).

GTM(영업)도 바뀐다 — **Ramp의 "forward-deployed engineers"** (자사 엔지니어를 고객사에 심어 맞춤 에이전트 구축)는, 병목이 모델이 아니라 **컨텍스트·통합**이기 때문에 나온 전략이다. 스타트업 시사점: "더 좋은 모델"이 아니라 **고객 조직의 컨텍스트를 축적·소유하고 통합에 깊이 박히는(sticky)** 제품이 해자와 예산을 얻는다. (유럽 주권 클라우드 논의는 그 컨텍스트를 *어디에 두느냐*의 규제 축.)

## 🔬 Enterprise Deep Dive — 에이전트 메모리 레이어 (self-hosted Knowledge Graph: cognee)

> 오늘의 1 Pick. "에이전트가 매번 백지에서 시작한다"는 Microsoft의 데이터 컨텍스트 문제를 정면으로 푸는 계층.

### 왜 쓰나 (도입 근거)
- **해결 문제**: LLM/에이전트는 세션마다 조직 맥락을 잊는다. 사람·프로젝트·정책의 **관계**를 모른 채 답하면 환각·오답. 메모리 레이어가 이 "백지 시작"을 없앤다.
- **대안 대비**: *단순 RAG*(문서 청크 검색, 엔티티·관계 없음) ↔ *지식그래프 메모리*(엔티티 정규화 + 관계 유지 → "이 이슈의 담당자와 관련 PR" 같은 다단계 회상). 규제 데이터면 **self-hosted**가 필수.
- **도입 효과**: 그라운딩 품질↑·환각↓, 크로스세션/크로스툴 연속성, 조직 지식의 자산화(락인 아닌 소유).
- **비용/리스크**: 엔티티 해소 오류(중복/오병합), 그래프 신선도(stale), 메모리 내 PII, 인프라(그래프+벡터 DB) 운영.

### 어떻게 연동하나 (기술 통합)
- **스택 위치**: `데이터 소스(메일·문서·이슈)` → `cognee(추출→그래프+벡터)` → `에이전트가 검색/회상(API·MCP)`. 에이전트 앱과 데이터 사이의 **컨텍스트 미들웨어**.
- **최소 통합(개념 예시)**:
```python
import cognee
await cognee.add(documents)     # 소스 인제스트
await cognee.cognify()          # 엔티티·관계 추출 → 지식그래프 + 벡터 구축
hits = await cognee.search("이 고객의 미해결 이슈와 담당자는?")  # 관계 기반 회상
# 에이전트는 hits를 컨텍스트로 그라운딩해 응답 (또는 MCP 툴로 노출)
```
- **운영 고려**: 증분 업데이트(변경분만 재인덱싱), 엔티티 해소 규칙, PII 마스킹·접근제어, 그래프/벡터 저장소 self-host, 회상 품질 평가(eval) 루프.
- **함정(gotchas)**: ① 쓰레기 인제스트 → 쓰레기 그래프(retrieval 품질이 07-10 TNS "retrieval이 정의적 과제"와 직결), ② 중복 엔티티 병합 실패, ③ 오래된 사실이 남아 오답.

### 도입 판단 체크리스트
- [ ] 에이전트가 **세션/도구를 넘는 조직 컨텍스트**를 필요로 하는가?
- [ ] 단순 문서검색(RAG)으로는 부족한 **관계형 질의**가 있는가?
- [ ] 데이터가 **사내에 있어야** 하는가(규제/기밀)?
- [ ] 인제스트·엔티티해소 **데이터 파이프라인을 운영**할 수 있는가?

→ 앞 2개가 Yes면 메모리 레이어 도입, 3번째가 Yes면 self-hosted(cognee류), 4번째가 No면 우선 관리형 메모리 SaaS로 시작.

## 💡 Deep Dive — 오늘의 인사이트

- **🧠 주요 개념** — *Agent Memory / Context Graph.* 에이전트를 "OS"로 올리는 결정적 부품은 모델이 아니라 **지속·관계형 컨텍스트**(cognee·지식그래프). RAG(검색)에서 Memory(관계 유지)로.
- **📦 주목 프로덕트** — *Skippr AI.* "제품 안의 AI 직원"이라는 포지션 자체가 메모리 없이는 성립 불가 — 컨텍스트 인프라가 곧 제품의 전제.
- **🎯 핵심 기획력** — *"도구"가 아니라 "OS/직원"으로 프레이밍.* Nautis(OS)·Skippr(직원)는 기능이 아니라 **조직의 운영 주체**로 스스로를 정의해 가치·예산의 크기를 키웠다.
- **💼 Tech → Business** — 해자가 *모델* 에서 **조직 컨텍스트 소유 + 통합 깊이**로 이동. GTM도 "forward-deployed engineers"(Ramp)처럼 컨텍스트·통합을 파고드는 방식으로 진화.
- **한 줄 종합:** 어제가 "모델을 사내에서 싸게 돌리기(추론)"였다면, 오늘은 **"에이전트에게 조직의 기억을 주기(컨텍스트)"** — 경쟁의 병목이 모델 성능에서 *지속 컨텍스트*로 확실히 넘어갔다.
